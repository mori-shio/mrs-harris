# Mrs. Harris 実装ルール

このドキュメントは、プロジェクト「Mrs. Harris」における実装時の重要なルールや方針をまとめたものです。エージェントはコードの修正や機能追加を行う際、これらのルールを厳守してください。

## 1. データベース・パフォーマンスに関するルール
- **レコードの連番管理（物理カラム化）**:
  ジョブの実行履歴 (`job_runs`) などの連番（N回目の実行）は、動的な `COUNT(*)` による算出ではなく、テーブルの物理カラム (`run_number` 等) として永続化すること。これにより Web 層での N+1 問題や都度カウントによるパフォーマンス悪化を防ぎます。
- **採番のアトミック性（トランザクション）**:
  新規レコード作成時に連番を振る場合は、競合（Race Condition）を防ぐため、必ずトランザクション (`pool.begin()`) 内で `SELECT COALESCE(MAX(run_number), 0) ... FOR UPDATE` のように行ロック（またはテーブルロック）を伴う方法で安全に採番を行うこと。
- **履歴・バージョン管理の徹底**:
  `job_runs` のような実行履歴データは、実行当時の設定バージョン（`job_history_id`）に必ず紐づくようにすること。`NULL` のまま登録されるようなデータ不整合を起こさない設計にしてください。
- **DB構造変更時の一貫性確保**:
  DB構造（スキーマ）の変更を行う場合は、関連するバックエンドのロジック、フロントエンドの表示処理、およびテストデータに至るまで、一貫して修正を適用し漏れがないようにすること。

## 2. Web/UI に関するルール
- **シェルコマンドの表示（アンラップ処理）**:
  バックエンドやワーカーに渡すシェルコマンドのペイロードとして `sh -c "..."` のようにラップして保存している場合、UI上（ジョブ詳細画面の「シェルコマンド」タブなど）でユーザーに表示する際は、`sh -c` を取り除き、ユーザーが実際に入力したスクリプト本体のみを表示すること。
- **動的カウントクエリの排除**:
  テンプレートレンダリング時に動的に `COUNT(*)` を発行してアイテム番号を算出するような処理は避け、すでに計算済みのカラム（`run_number`等）を直接利用すること。

## 3. その他の規約
- **Tera テンプレート**:
  フロントエンドの描画には Tera テンプレートエンジンを使用。UIコンポーネントを追加する際は既存のデザインシステム・スタイル（CSS）を踏襲し、リッチで直感的なUXを維持すること。
- **エラーハンドリング**:
  `anyhow` などを活用し、エラー発生時は握り潰さず適切に伝播（Propagate）またはログ出力すること。
- **エージェントのサーバー実行と完了報告**:
  `cargo run` やサーバープロセスを起動する際は、エージェント自身のタスクが UI 上で永続的に「task running」状態として残らないよう、`nohup ... > server.log 2>&1 &` などの手法を用いて完全にバックグラウンドプロセスとして切り離すこと。
  また、起動コマンドを実行した直後に完了報告をするのではなく、必ず `tail -n 20 server.log` 等で起動ログを最後まで確認し、パニックやエラーが発生せずに正常稼働（`listening on ...` など）していることを確証した上で完了報告を行うこと。

## 4. UI確認リストと検証フロー
- **UI要件の管理**:
  ユーザー様から「XX画面のXXはXXにしたい」などのUIに関する要望を受けた場合、実装を行う前に必ず `docs/ui_checklists/` 配下の該当画面のチェックリストファイル（例: `job_edit.md`）へ要件を箇条書きで追記すること。
  （どの画面のファイルを参照すべきかは `docs/ui_checklists/MANIFEST.md` を確認する）
- **実装完了後の検証と報告**:
  サーバー再起動後、修正内容に関連する画面に絞ってチェックリストの項目を検証すること。
  検証時はトークン消費を抑えるためフルページのHTML取得などは避け、Chrome DevTools MCP によるピンポイントなDOM検証や最小限のスクリプトを用いること。
  対象画面のチェックリスト項目のパスが確認できてから、ユーザー様へ完了報告を行うこと。
