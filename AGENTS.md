# Mrs. Harris 実装ルール

このドキュメントは、プロジェクト「Mrs. Harris」における実装時の重要なルールや方針をまとめたものです。エージェントはコードの修正や機能追加を行う際、これらのルールを厳守してください。

また、コーディングを始める前に、context7（MCPサーバ）を使って最新のドキュメントを取得してください。

## 1. データベース・パフォーマンスに関するルール
- **レコードの連番管理（物理カラム化）**:
  ジョブの実行履歴 (`job_runs`) などの連番（N回目の実行）は、動的な `COUNT(*)` による算出ではなく、テーブルの物理カラム (`run_number` 等) として永続化すること。これにより Web 層での N+1 問題や都度カウントによるパフォーマンス悪化を防ぎます。
- **採番のアトミック性（トランザクション）**:
  新規レコード作成時に連番を振る場合は、競合（Race Condition）を防ぐため、必ずトランザクション (`pool.begin()`) 内で `SELECT COALESCE(MAX(run_number), 0) ... FOR UPDATE` のように行ロック（またはテーブルロック）を伴う方法で安全に採番を行うこと。
- **履歴・バージョン管理の徹底**:
  `job_runs` のような実行履歴データは、実行当時の設定バージョン（`job_history_id`）に必ず紐づくようにすること。`NULL` のまま登録されるようなデータ不整合を起こさない設計にしてください。
- **DB構造変更時の一貫性確保**:
  DB構造（スキーマ）の変更を行う場合は、関連するバックエンドのロジック、フロントエンドの表示処理、およびテストデータに至るまで、一貫して修正を適用し漏れがないようにすること。

## 2. Web/UI に関するルール
- **シェルコマンドの表示（アンラップ処理）**:
  バックエンドやワーカーに渡すシェルコマンドのペイロードとして `sh -c "..."` のようにラップして保存している場合、UI上（ジョブ詳細画面の「シェルコマンド」タブなど）でユーザーに表示する際は、`sh -c` を取り除き、ユーザーが実際に入力したスクリプト本体のみを表示すること。
- **動的カウントクエリの排除**:
  テンプレートレンダリング時に動的に `COUNT(*)` を発行してアイテム番号を算出するような処理は避け、すでに計算済みのカラム（`run_number`等）を直接利用すること。

## 3. 実装時の基本規約
- **Askama テンプレート**:
  フロントエンドの描画には Askama テンプレートエンジンを使用。UIコンポーネントを追加する際は既存のデザインシステム・スタイル（CSS）を踏襲し、リッチで直感的なUXを維持すること。
- **エラーハンドリング**:
  `anyhow` などを活用し、エラー発生時は握り潰さず適切に伝播（Propagate）またはログ出力すること。
- **最新ドキュメント確認**:
  コーディングを始める前に、変更対象の主要ライブラリやフレームワークについて context7（MCPサーバ）を使って最新のドキュメントを取得すること。対象が明確でない文書修正のみの場合は、リポジトリ内の現行コード・設定を一次情報として扱ってよい。
- **検証可能な最小変更**:
  変更は要件に必要な範囲へ絞ること。既存の未コミット変更はユーザーの作業として扱い、明示指示なしに巻き戻さないこと。
- **1ラリーごとのコミット**:
  ユーザーからの1回の依頼（1ラリー）に対する作業が完了したら、検証結果を確認したうえで必ず git commit を作成すること。コミット前には `git status` と `git diff` で差分を確認し、今回の依頼に関係する変更だけを staging すること。未追跡ファイルや既存の未コミット変更がある場合は、ユーザーの作業を混ぜ込まないよう明示的に除外すること。

## 4. UI確認リストと検証フロー
- **UI要件の管理**:
  ユーザー様から「XX画面のXXはXXにしたい」などのUIに関する要望を受けた場合、実装を行う前に必ず `docs/ui_checklists/` 配下の該当画面のチェックリストファイル（例: `job_edit.md`）へ要件を箇条書きで追記すること。
  （どの画面のファイルを参照すべきかは `docs/ui_checklists/MANIFEST.md` を確認する）
- **UI実装後の確認対象**:
  修正後は、変更内容に関連する画面だけを対象に検証すること。全画面巡回やフルページHTML取得は避け、該当チェックリストの項目を満たす最小範囲で確認すること。
- **Codex in-app browser の優先利用**:
  UIの動作確認では、Codex の Browser / in-app browser を優先して使用すること。対象URLへ移動し、URL・タイトル・主要DOM要素・表示テキスト・リンク先・フォーム状態・モーダル表示など、要件に直結する信号をピンポイントに確認すること。
- **DOM検証の粒度**:
  検証時は `document.body.innerText` 全量取得やフルHTML取得を避けること。対象コンテナ、ボタン、リンク、入力欄、テーブル行などに絞った `querySelector` / locator / 最小スクリプトで確認すること。
- **視覚確認が必要な場合**:
  レイアウト崩れ、重なり、余白、色、モーダル、レスポンシブ表示など DOM だけでは判断しにくい変更では、in-app browser のスクリーンショットまたは表示確認を行うこと。必要に応じて desktop/mobile の代表的な viewport で確認すること。

## 5. 実装後の検証コマンド
- **Rust静的検証**:
  コード変更後は原則として `cargo fmt --check --all`、`cargo check --workspace` を実行すること。ロジック変更やDB/スケジューラ/workerに関わる修正では `cargo test --workspace` も実行すること。
- **Clippy**:
  共有ロジック、DB、API、worker、または広範囲の変更では `cargo clippy --workspace -- -D warnings` を実行すること。既存警告がある場合は、今回の変更で増やしていないことを明示すること。
- **DB変更の検証**:
  migration を追加・変更した場合は、関連する Rust のDBアクセス、テンプレート表示、テストデータ、UI表示の整合性を確認すること。`job_runs` のような履歴データでは、採番・履歴ID・表示番号の一貫性を必ず確認すること。

## 6. サーバー起動とBrowser確認
- **サーバープロセスの切り離し**:
  `cargo run` やサーバープロセスを起動する際は、エージェント自身のタスクが UI 上で永続的に「task running」状態として残らないよう、`nohup ... > server.log 2>&1 &`、`screen -dmS ...` などの手法でバックグラウンドプロセスとして切り離すこと。
- **起動確認**:
  起動コマンド直後に完了報告しないこと。`tail -n 20 server.log`、`lsof -nP -iTCP:8080 -sTCP:LISTEN`、または Browser での `http://localhost:8080/` 到達確認により、パニックや起動失敗がないことを確認すること。
- **Browser検証の必須化**:
  UI変更を含む場合は、サーバー起動後に in-app browser で対象画面を開き、該当チェックリスト項目を確認してから完了報告すること。検証対象は変更箇所に絞り、過剰なページ探索は行わないこと。
- **完了報告に含める証跡**:
  完了報告には、実行した検証コマンド、Browserで確認したURL、確認したチェックリスト項目、未確認事項があればその理由を含めること。
