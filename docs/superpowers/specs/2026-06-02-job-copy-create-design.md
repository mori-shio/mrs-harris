# ジョブ一覧: 既存ジョブをコピーして新規作成する導線

## 背景

ジョブ一覧画面には既存の `新規ジョブ作成` 導線があるが、既存ジョブをベースに似た設定のジョブを作る需要に対しては再入力コストが高い。  
今回追加したいのは、一覧画面から既存ジョブを選び、その設定を新規作成フォームへ初期値として流し込む `コピーして作成` 導線である。

## 目標

- ジョブ一覧画面から `コピーして作成` を開始できること
- コピー元ジョブを名前の前方一致で絞り込めること
- コピー元を選ぶと新規ジョブ作成画面へ遷移し、設定がプリセットされること
- 新規ジョブとして保存される前提なので、ジョブ名や履歴/実行履歴のような識別情報はコピーしないこと

## 非目標

- サーバーサイド検索 API の追加
- 実行履歴、履歴バージョン、ID の複製
- ジョブ作成画面の新しい保存フロー追加
- コピー元との差分表示

## UX

### ジョブ一覧画面

- `新規ジョブ作成` ボタンの横に `コピーして作成` ボタンを追加する
- 押下時にモーダルを表示する
- モーダル内には以下を置く
  - 検索 input
  - 候補リスト
  - 閉じる導線

### 検索

- 検索対象はジョブ名のみ
- マッチ条件は前方一致
- モーダルを開いた時点でジョブ一覧データをクライアントへ持ち、フロント側でフィルタする
- 空文字時は全件表示でよい

### 候補表示

- 最低限 `ジョブ名` を表示
- 余裕があれば短い説明 1 行を添える
- 候補クリックで `/jobs/new?copy_from=<job_name>` へ遷移する

## データフロー

### 一覧画面

- `jobs_page` が描画時に、モーダル用のコピー元候補をテンプレートへ渡す
- 候補は最低限 `name` と `description` を持つ
- テンプレート内 JS で前方一致フィルタを行う

### 新規作成画面

- `new_job_page` は `copy_from` query parameter を受け取る
- `copy_from` がある場合:
  1. `get_job_by_name` でコピー元ジョブを取得
  2. 既存のフォーム初期値構築ロジックへ流し込む
  3. `name` は空文字へ上書きする
- `copy_from` がない場合は従来どおり空フォーム

## コピー対象

- 説明
- ジョブタイプ
- ワーカー定義
- スペース
- スケジュール
- シェルスクリプト
- DAG 定義
- 直接設定環境変数
- SSM パラメータ設定
- タイムアウト
- リトライ設定
- タグ
- Slack 通知設定
- 有効/無効

## コピーしないもの

- ジョブ名
- job id
- 作成日時/更新日時
- job_history
- job_runs
- job_history_id
- 実行番号や各種履歴メタデータ

## 実装方針

### テンプレート

- `jobs/list.html` に `コピーして作成` ボタンとモーダルを追加
- モーダルは既存のアプリ内モーダルの見た目を踏襲する
- 候補リストはカード一覧ではなく、検索用の縦リストで十分

### バックエンド

- `JobsListTemplate` にコピー候補リストを追加
- `new_job_page` で `copy_from` を受ける
- 既存の `edit` 系初期化処理と重複する項目は、可能なら小さな helper へ寄せる
- ただし今回の変更範囲は限定し、不要な大規模抽象化は行わない

### エラーハンドリング

- `copy_from` に該当ジョブがない場合:
  - 空フォームへ silently fallback しない
  - `404` または分かるエラー応答にする
- これにより誤ったジョブ名指定を見逃さない

## テスト/確認

### Rust

- `cargo fmt --check --all`
- `cargo check --workspace`
- `cargo test --workspace` はロジック変更が入るため実行対象

### UI

対象画面のみ確認する。

- `/jobs`
  - `コピーして作成` ボタンがある
  - モーダルが開く
  - 前方一致で絞り込まれる
  - 候補クリックで `/jobs/new?copy_from=...` へ遷移する
- `/jobs/new?copy_from=<job>`
  - ジョブ名だけ空
  - その他の項目がコピー元に揃う

## 採用理由

フロント内フィルタ方式を採用する理由は以下。

- 追加 API が不要
- 一覧画面の責務から大きく外れない
- 即時反応で UX が軽い
- 今回の課題は「検索基盤」ではなく「コピー導線」だから

ジョブ数が将来大きく増えてクライアント保持が重くなる場合は、その時点でサーバー検索 API へ切り替える。
